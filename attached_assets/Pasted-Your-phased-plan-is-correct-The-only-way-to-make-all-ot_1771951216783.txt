Your phased plan is correct. The only way to make “all other states are data entry” true is to make the compliance matrix the single source of truth and to enforce a strict rule: engine code cannot branch on state/form IDs (only on matrix-provided enums and rules).

Below is a concrete, buildable database design + admin tooling surface area + engine contract that supports Michigan “official PDF overlay” without any if (state === 'MI').

Phase 1 — Compliance Matrix Schema (PostgreSQL)
1) Core reference tables

states

id (PK)

code (e.g., MI, CA)

name

forms

id (PK)

local_overlay_risk (low|med|high)

disclaimer_text (required for med/high)

form_versions (your “statute snapshot + effective dates + locking” lives here)

id (PK)

approval_notes

Phase 2 — Admin Tooling (UI)

Minimum screens (in this order):

Forms

Create form (state, name, key, category, overlay risk, disclaimer)

List forms + active version + status

Form Versions

Create draft version, set effective dates, status workflow (draft → approved)

Paste statute snapshot + citation + retrieved timestamp

“Mark Review Required” (locks generation)

Day Rules

Select DayType + notice period default + counting convention

Attach HolidayCalendarRef (only if DayType ≠ calendar)

Wizard Fields

Add/edit fields, types, required flags, help text

Add validations (rules + messages)

Configure conditional visibility and gating questions

Service Rules

Toggle allowed methods

Configure hierarchy (prior attempts)

Configure combined requirements (additional methods)

Configure acknowledgments

Configure lease gates (TX-style) as generic “gates”

Language Blocks

Maintain canonical blocks

Attach blocks per form_version by type (warning/options/cure/etc.)

Output Templates

Choose mode

Overlay: upload base PDF + set overlay fields (page/x/y/font)

Formatted: edit HTML template and DOCX template mapping

Preview / Test Harness

Enter sample inputs → generate PDF preview + show:

validations

calculated dates

service method enforcement

language blocks inserted

Show “diff” between versions for audit/QA

Phase 3 — Matrix-Driven Engine Contracts (No state branching)
Engine modules

Matrix Resolver

Input: form_key + as_of_date

Output: fully hydrated FormDefinition object (version + fields + rules + templates)

Validation Engine

Runs field_validations, gate rules, and service constraints from matrix

Date Engine

Input: day_type, notice_period_days, holiday_calendar_dates, service_date, counting_convention

Output: compliance_deadline, earliest_filing_date, explain_formula string

Service Engine

Renders available service methods based on matrix + user’s gate answers

Enforces hierarchy + combined requirements + acknowledgments

Renderer

If mode = official_pdf_overlay: overlay engine places fields by overlay_fields

If mode = leaseshield_formatted: HTML→PDF + DOCX template merge

Hard rule to enforce internally

No code is allowed to query state_code to branch behavior.

All behavior is derived from:

day_type

holiday_calendar_ref

service rules graph

field validations

output_template.mode

Phase 4 — Michigan Validation (DC 100a / DC 100c)

Michigan passes when:

The only data you enter is:

compliance matrix records + overlay coordinates + required blocks

The engine:

validates required fields

computes dates (calendar days)

renders service certificate correctly (allowed methods)

outputs a PDF that matches the SCAO layout because the overlay template tells it how

That proves the architecture.

One implementation tip that prevents pain later

Treat overlay coordinate mapping as its own “mini-product”:

add snap-to-grid

page preview with click-to-place fields

per-field test values

export/import overlay maps (JSON)

Michigan will be fast if you can place 20–40 overlay fields efficiently.
