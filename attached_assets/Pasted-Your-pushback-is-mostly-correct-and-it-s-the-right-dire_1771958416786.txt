Your pushback is mostly correct, and it’s the right direction operationally. The key is to formalize the routing contract so “official overlay” always uses the correct path, whether the PDF is AcroForm-fillable or requires coordinate overlays—without reintroducing state branching or title heuristics.

Below is the tightened plan (and the exact Replit instructions) that incorporates your points and closes the remaining gaps.

1) Where your pushback is right (and what to codify)
A) Prefer AcroForm field filling over x/y overlays

Yes. If the official PDF has interactive fields, the primary render strategy should be:

fill AcroForm fields (text + checkboxes)

flatten

output

Coordinate overlay is the fallback when:

the PDF has no form fields, or

field names are inconsistent/unreliable, or

checkboxes aren’t representable cleanly

Codify this as: render_strategy = form_fields | coordinates inside the output_templates record (or derived by inspection + override).

B) Keep output mode in output_templates, not templates

Agreed. Do not force every library template to declare an output mode.

What you do need is a deterministic, explicit link from a “template” to a “notice form” (or “output template config”) so the generator can route without heuristics.

Codify this as: templates.notice_form_key (nullable) OR templates.output_template_id (nullable). Either is fine; I’d choose output_template_id if it’s truly 1:1.

C) The real offender is documentGenerator.ts + routes.ts shortcuts

Also correct. Rendering pipeline may be mostly there; the bug is the call sites that:

add LeaseShield headers/metadata

merge lease content

have Michigan-only shortcuts

contain state conditionals

Codify this as: the only module allowed to generate PDFs is a new DocumentRenderer service. No route/component may bypass it.

2) The hard requirement you should add (to prevent regressions)

“Official overlay PDFs must be byte-structure derived from the base PDF, not rebuilt via HTML.”

Concrete enforcement:

If output_templates.mode = official_pdf_overlay, the renderer must:

load base pdf bytes

fill fields / draw overlays

flatten

save

It must never call Puppeteer/HTML for that document.

That single constraint eliminates the MI “still doesn’t look right” issue.
